# Token 使用效率技能（預設啟用）

## 核心原則

模型無法控制 input context（系統會自動累積對話歷史與工具結果），但可以控制**自己的輸出**與**工具呼叫行為**。

## Output 精簡原則

### 回覆時

- 直接回答，不加前言後語（「好的」、「讓我...」、「我來分析一下...」）
- 一句話能說完就不要寫三段
- 不重複使用者已經知道的資訊（如「如您所見...」、「根據上面的結果...」）
- 不替程式碼加無謂的註解說明
- 不總結自己做過了什麼（除非使用者問）
- 不輸出感謝、禮貌用語

### 檔案操作時

- 寫入/編輯檔案時不附帶說明文字
- diff 只貼必要部分，不貼整份檔案
- 工具 `description` 參數保持簡短（5-10 字）

### 錯誤處理時

- 錯誤訊息摘要關鍵行號與類型，完整 stack trace 只在使用者要求時才貼
- 不重複描述錯誤原因（使用者已經看到錯誤訊息）

## 工具呼叫效率

- 優先使用更精準的工具：用 `grep` 取代多次 `read`，用 `glob` 取代 `ls`
- 不要為了找一個關鍵字而讀取整份檔案
- 同一批獨立操作合併在同一個 message 中平行呼叫
- 不要在搜尋後又對已知結果重複搜尋
- 避免連續的小量 read（每次 30 行），一次讀取夠用的範圍

### 檔案過濾（避免浪費 token 在無意義的檔案上）

- 執行 `grep` 前先用 `glob` 確認目標檔案類型，避開 `.min.js`、`.bundle.js`、`.chunk.js` 等打包檔
- 對 `.svg`、`.png`、`.ico`、`.woff2` 等二進位/非文字檔不要下 `grep` 或 `read`
- 對 `dist`、`build`、`.next`、`Pods` 等目錄原則上不搜尋；`node_modules` 只有在任務明確涉及修改或檢查指定套件時才搜尋
- 若檔案超過 500 行且只需要特定函式，先用 `grep` 定位行號再用 `read offset` 精準讀取

## 上下文管理（實際可行的）

| 項目 | 做法 |
|------|------|
| 已完成任務 | 從 todo 清單移除，不保留詳細描述 |
| 舊版檔案內容 | 不主動重提，除非使用者問 |
| 偏離主題的討論 | 不延續、不總結、不引用 |
| 長日誌/輸出 | 摘要 1-3 行核心訊息 |

## 使用方式

**全程啟用** - 每一輪回覆都應遵循上述原則，無需等待系統指示。

強化模式關鍵字（當使用者明確要求時請更嚴格遵守）：
- 「省 token」、「精簡回覆」、「減少輸出」、「不要廢話」
